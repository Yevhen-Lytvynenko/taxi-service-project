# Ролі і доступи адмін-панелі

Документ описує цільову модель ролей для адмін-панелі (`packages/web-admin`) і
правила обмежень на фронтенді та бекенді. Стан — **проектний**: реалізація
очікує на винос поля `User.role` у окрему сутність `Role` (many-to-many через
`UserRole`). До цього моменту всі акаунти з `role === 'ADMIN'` мають
повний доступ, решта — тільки перегляд мобільних застосунків.

Пов’язані точки в коді:

- Меню — `packages/web-admin/src/components/organisms/Layout.tsx`, три секції: `Операції (ІС)` → `Аналітика (демо)` → `Налаштування`.
- Бекенд — `packages/backend/src/middleware/authorize.middleware.ts`, перевіряє `req.user.role` для захищених маршрутів.
- UI сторінка — `packages/web-admin/src/pages/settings/RolesSettings.tsx` (матриця ролі × сторінки).

## Перелік ролей

| Код | Назва | Основне призначення |
|---|---|---|
| `ADMIN` | Адміністратор | Повний доступ, керування ролями і налаштуваннями |
| `MANAGER` | Менеджер операцій | Щоденні операції: клієнти, водії, замовлення, відгуки |
| `ACCOUNTANT` | Бухгалтер | Фінанси, транзакції, тарифи, фінансова аналітика |
| `DISPATCHER` | Диспетчер | Замовлення, чати, стежування, водії (тільки читання) |
| `AUDITOR` | Аудитор | Повний read-only + експорт, без права змінювати дані |

## Правила обмежень

### Фронтенд

1. **Меню**: пункти з доступом `немає` приховуються у `Layout.tsx` на основі масиву дозволів поточного користувача.
2. **Кнопки дій** у `AppDataGrid` і формах:
   - `onCreate` і `buildRowActionsColumn({ onEdit, onDelete })` передаються тільки якщо користувач має `повний` доступ.
   - Експорт у CSV/XLSX/JSON дозволений для `читання` і `повний` (аудитори можуть вивантажувати дані).
3. **Маршрути**: захищений `Layout` рендерить `Navigate('/forbidden')` для ролей без доступу до конкретного шляху.
4. **Колонки з персональними даними** (телефони, email, суми виплат) прикриваються на рівні `columnVisibilityModel` за замовчуванням для ролей `DISPATCHER`, `AUDITOR`.

### Бекенд

1. `authorize.middleware.ts` отримує масив дозволених ролей і повертає `403` у випадку невідповідності.
2. Усі write-ендпоінти (`POST`/`PUT`/`DELETE`) окремо вимагають `ADMIN`, `MANAGER` або профільну роль згідно з матрицею.
3. Фінансові ендпоінти (`/transactions`, `/tariffs`) — доступні лише `ADMIN` і `ACCOUNTANT` на запис; інші ролі — `GET` з маскуванням (для `DISPATCHER` повертаємо без `amount`).
4. Аудит дій (майбутнє): кожна мутація пише запис у `AuditLog(userId, action, entity, entityId, diff, ip, at)`.

## План міграції

1. Додати в `schema.prisma`:
   - модель `Role` (id, code, name, description),
   - модель `UserRole` (userId, roleId, assignedAt, assignedBy),
   - колонку `User.primaryRoleId` для зворотної сумісності з існуючим `role`.
2. Seed ролей: `ADMIN`, `MANAGER`, `ACCOUNTANT`, `DISPATCHER`, `AUDITOR` (+ `CLIENT`, `DRIVER` як системні, нередаговані).
3. Оновити `authorize.middleware.ts` — читати `req.user.roles: string[]` і перевіряти перетин.
4. Додати ендпоінт `GET /me/permissions`, який повертає набір дозволів для фронтенду.
5. На фронтенді — хук `usePermissions()` і компонент `<RequireRole roles={["ADMIN","MANAGER"]}>`.
6. Реалізувати CRUD на сторінці `Налаштування → Ролі`: призначення/зняття ролей для конкретного користувача.

## Додаткові ідеї для подальших ітерацій

- **Журнал аудиту** (окрема сторінка, read-only, з фільтрами по користувачу/типу дії/даті).
- **Bulk-операції** у `AppDataGrid` (checkbox-selection + масове видалення/експорт).
- **Збережені пресети** фільтрів/колонок з власною назвою.
- **Глобальний пошук** у шапці (по замовленнях, клієнтах, водіях).
- **Сповіщення в шапці** (live `socket`: нове замовлення, скарга, помилка оплати).
- **Тема і локалізація** (UA/EN/RU).
- **API-ключі і вебхуки** — секція «Налаштування».
- **Експорт PDF-звітів** для фінансів (через `jspdf` + `jspdf-autotable`, уже в залежностях).
